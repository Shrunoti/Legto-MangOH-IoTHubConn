1 /** @file smack.h
2  *
3  * Simplified Mandatory Access Control Kernel (SMACK) provides a simple solution
4  * for mandatory access control (MAC). MAC provides the ability for a centralized entity to
5  * set access policy for system resources.
6  *
7  * Linux's default access control policy is governed by permission bits on system resources
8  * (files, directories, devices, etc.). Permission bits can be modified by the resource owner
9  * (process with the same user ID as the resource). The access control policy is at
10  * the discretion of the resource owner; this system is classified as DAC (discretionary
11  * access control). With DAC, policies are set in a distributed manner as there
12  * are often many system users, each setting the access policy for its own resources.
13  *
14  * In contrast, MAC policies are set for all system resources by a centralized entity.
15  *
16  * Linux's DAC has known weaknesses that can lead to security leaks. MAC is often used to
17  * overcome some of the short comings of DAC for systems that require a higher level of security.
18  *
19  * SMACK isn't the only MAC solution available. Because it's a simple solution, it's not flexible enough
20  * to handle all use cases. For the majority of use cases, it will be easier to setup and maintain.
21  *
22  * SMACK supplements Linux's DAC system. DAC permissions are checked first; if access is granted,
23  * SMACK permissions are then checked. Consequently, SMACK can only limit access,
24  * it can't grant access beyond DAC permissions.
25  *
26  * SMACK uses 'labels' on resources (objects in SMACK terminology) and processes (subjects) to
27  * determine access. Labels on resources can only be set by a privileged process. A privileged
28  * process can only set it's own label but not labels of other processes.
29  *
30  * There are a number of single character labels ("_", "^", "*", "?", "@") that have special
31  * meanings. SMACK restricts read/write/execute access based on the subject label and the object
32  * label according to the following rules:
33  *
34  * 1. Any access requested by a task labelled "*" is denied.
35  * 2. A read or execute access requested by a task labelled "^" is permitted.
36  * 3. A read or execute access requested on an object labelled "_" is permitted.
37  * 4. Any access requested on an object labelled "*" is permitted.
38  * 5. Any access requested by a task on an object with the same label is permitted.
39  * 6. Any access requested that is explicitly defined in the loaded rule set is permitted.
40  * 7. Any other access is denied.
41  *
42  * Rule 6 lets us use explicit rules through adding specific labels. Explicit rules
43  * define the access rights a subject label can have on an object label. See the
44  * @ref c_smack_setRules section for details. Only privileged processes can set rules.
45  *
46  * @section c_smack_privProcess Privileged Processess
47  *
48  * Privileged processes use the CAP_MAC_OVERRIDE capability. It's also possible to configure
49  * the system so the CAP_MAC_OVERRIDE is honoured only for processes with a specific label. This
50  * configuration allows the system to restrict root processes (have CAP_MAC_OVERRIDE) that don't
51  * have the proper SMACK label.
52  *
53  * @section c_smack_assignLabels Assigning Labels
54  *
55  * Use smack_SetMyLabel() to set the SMACK label for the calling process. The calling process must
56  * be a privileged process. Setting SMACK labels for other processes isn't possible.
57  *
58  * To set the SMACK label for file system objects use smack_SetLabel(), again the calling process
59  * must be privileged.
60  *
61  * @section c_smack_setRules Setting Rules
62  *
63  * Use smack_SetRule() to set an explicit SMACK rule that gives a specified subject access to a
64  * specified object.
65  *
66  * Copyright (C) Sierra Wireless Inc.
67  */

162 /**
163  * Sets an explicit SMACK rule.
164  *
165  * An explicit SMACK rule defines a subject's access to an object. The access mode can be any
166  * combination of the following.
167  *
168  * a: indicates that append access should be granted.
169  * r: indicates that read access should be granted.
170  * w: indicates that write access should be granted.
171  * x: indicates that execute access should be granted.
172  * t: indicates that the rule requests transmutation.
173  * -: is used as a place holder.
174  *
175  * For example:
176  * "rx" means read and execute access should be granted.
177  * "-" means that no access should be granted.
178  *
179  * @note If there is an error this function will kill the calling process.
180  */
