6  * @ref c_cgrp_settingAttributes <br>
7  * @ref c_cgrp_addingProcesses <br>
8  * @ref c_cgrp_delete <br>
9  * @ref c_cgrp_threadSafety <br>
10  *
11  *
12  * Cgroups, short for control groups, is a Linux kernel feature that allows hierarchal groupings of
13  * processes. Each group can then be configured with specific attributes that apply to the entire
14  * group. Control groups should not be confused with process groups as this is a different concept.
15  *
16  * A cgroup can contain other sub-groups and can be arranged in a tree structure similar to
17  * directories in a file system. However, unlike a file system cgroups can have multiple roots.
18  * These separate cgroup trees are called hierarchies. For example, a system's cgroups could be
19  * arranged in the following manner:
20  *
21  * @verbatim
22  Hierarchy1 Hierarchy2
23  / \ / \
24  group1 group2 group1 group2
25  / \ \
26  subgroup1 subgroup2 subgroup1
27 @endverbatim
28  *
29  * Attributes that a cgroup can have are available through kernel sub-systems. For example,
30  * the memory sub-system can be used to set the memory limit for all processes in a cgroup. A
31  * hierarchy must have at least one sub-system attached to it and in general a sub-system can only
32  * be attached to a single hierarchy.
33  *
34  * A process can only be a part of one cgroup in each hierarchy. Having separate hierarchies allows
35  * for more flexible control of cgroups.
36  *
37  * In practice cgroups are used mainly for limiting system resources.
38  *
39  *
40  * @section c_cgrp_layout Hierarchy Layout
41  *
42  * In this implementation of cgroups each sub-system is attached to its own hierarchy. In other
43  * words there is a one-to-one mapping of hierarchy and sub-systems so the terms hierarchy and
44  * sub-system will be used interchangeably henceforth.
45  *
46  *
47  * @section c_cgrp_init Initialization
48  *
49  * On system start-up the cgrp_Init() function must be called to setup the hierarchies. Cgroups are
50  * by default non-persistent so cgrp_Init() must called every time the system starts.
51  *
52  *
53  * @section c_cgrp_create Creating cgroups
54  *
55  * To create a cgroup for a sub-system call cgrp_Create().
56  *
57  *
58  * @section c_cgrp_settingAttributes Setting cgroup Attributes
59  *
60  * Cgroups created for a specific sub-system can only set attributes specific to that sub-system.
61  * For example:
62  *
63  * @code
64  * // cgroup created for the cpu sub-system.
65  * cgrp_Create(CGRP_SUBSYS_CPU, "MyApp");
66  *
67  * // cgroup created for the memory sub-system with the same name. This is a separate cgroup
68  * // but it can have the same name because it is in a different hierarchy.
69  * cgrp_Create(CGRP_SUBSYS_MEM, "MyApp");
70  *
71  * // Set the cpu share for the cgroup in the cpu sub-system to half of the default value.
72  * cgrp_cpu_SetShare("MyApp", 512);
73  *
74  * // Set the memory limit for the cgroup in the memory sub-system.
75  * cgrp_mem_SetLimit("MyApp", 100);
76  * @endcode
77  *
78  *
79  * @section c_cgrp_addingProcesses Adding Processes to a cgroup
80  *
81  * Processes can be added to a cgroup, by PID, using cgrp_AddProc(). If a process already belonging
82  * to a cgroup is added to another cgroup in the same hierarchy, the process is moved but not copied
83  * to the second cgroup, because processes can only be in one cgroup per hierarchy.
84  *
85  * Processes that are forked by other processes always inherit the cgroup of their parent.
86  *
87  * When a process dies it is automatically removed from all cgroups it belongs to.
88  *
89  * @section c_cgrp_delete Deleting cgroups
90  *
91  * To delete a cgroup call cgrp_Delete(). Cgroups can only be deleted if they do not contain any
92  * processes.
93  *
94  *
95  * @section c_cgrp_threadSafety Thread Safety
96  *
97  * The functions in this API are not thread safe. Other synchronization methods must be used to
98  * control concurrent access to the cgroups.
99  *
100  * Copyright (C) Sierra Wireless Inc.
101  */

299 /**
300  * Sets the cpu share of a cgroup.
301  *
302  * Cpu share is used to calculate the cpu percentage for a process relative to all other processes
303  * in the system. Newly created cgroups and processes not belonging to a cgroup are given a default
304  * value of 1024. The actual percentage of the cpu given to a process is calculated as:
305  *
306  * (share value of process) / (sum of shares from all processes contending for the cpu)
307  *
308  * All processes within a cgroup share the available cpu share for that cgroup.
309  *
310  * For example:
311  *
312  * cgroupA is configured with the default share value, 1024.
313  * cgroupB is configured with 512 as its share value.
314  * cgroupC is configured with 2048 as its share value.
315  *
316  * cgroupA has one process running.
317  * cgroupB has two processes running.
318  * cgroupC has one process running.
319  *
320  * Assuming that all processes in cgroupA, cgroupB and cgroupC are running and not blocked waiting
321  * for some I/O or timer event and that another system process is also running.
322  *
323  * Sum of all shares (including the one system process) is 1024 + 512 + 2048 + 1024 = 4608
324  *
325  * The process in cgroupA will get 1024/4608 = 22% of the cpu.
326  * The two processes in cgroupB will share 512/4608 = 11% of the cpu, each process getting 5.5%.
327  * The process in cgroupC will get 2048/4608 = 44% of the cpu.
328  * The system process will get 1024/4608 = 22% of the cpu.
329  *
330  * @return
331  * LE_OK if successful.
332  * LE_FAULT if there was an error.
333  */
334 //--------------------------------------------------------------------------------------------------
335 le_result_t cgrp_cpu_SetShare
336 (
337  const char* cgroupNamePtr, ///< [IN] Name of the cgroup to set the share for.
338  size_t share ///< [IN] Share value to set. See the function header for more
339  /// details.
340 );
